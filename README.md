# Yandex 360 Mail Routing: forward inbound писем с “опасными” вложениями

Скрипт `add_rules.py` добавляет новое правило маршрутизации почты в **Yandex 360** (Admin API), которое **отлавливает входящие письма** с вложениями определённых расширений и **пересылает** такие письма на заданный email.

Скрипт:

* Получает текущий список routing-правил организации через API.
* Генерирует одно новое правило с условием `"$or"` по списку расширений.
* Добавляет правило в конец списка (старые правила **не удаляются**).
* Отправляет обновлённый список правил обратно в API. 

## Что именно делает правило

Создаётся правило для `scope.direction = inbound`:

* **Condition**: вложение, где `attach:filename` содержит `.<ext>` для каждого `ext` из списка `EXTENSIONS` (условия объединены через `"$or"`).
* **Action**: `forward` на адрес `TARGET_EMAIL`.
* **terminal = False** (то есть после срабатывания правила обработка цепочки правил **может продолжаться**, если дальше есть другие правила). 

## Требования

* Python 3.8+ (подойдёт и новее)
* Библиотека `requests`
* Доступ к Yandex 360 Admin API:

  * `ORG_ID` вашей организации
  * OAuth-токен с правами на управление mail routing rules (уровень прав зависит от вашей конфигурации в Yandex 360)

## Установка

```bash
git clone <repo>
cd <repo>
python3 -m venv .venv
source .venv/bin/activate
pip install -U pip
pip install requests
```

## Настройка

Откройте `add_rules.py` и задайте значения в блоке **ВАШИ ДАННЫЕ**:

* `ORG_ID` — идентификатор организации в Yandex 360
* `TOKEN` — OAuth токен
* `TARGET_EMAIL` — почта, на которую будет выполняться пересылка (куда уходят письма с вложениями из списка)

Список расширений находится в `EXTENSIONS` — при необходимости его можно отредактировать. 

## Запуск

```bash
python3 add_rules.py
```

Ожидаемое поведение в консоли:

1. Получение текущих правил
2. Генерация нового правила
3. Добавление правила в список
4. PUT обновлённых правил

При успехе скрипт выводит:

* `✅ Успех! Новое правило успешно добавлено, старые сохранены.` 

## Важно про идемпотентность (повторные запуски)

Скрипт **всегда добавляет новое правило**, не проверяя, существует ли уже аналогичное.
Если запустить несколько раз — в правилах появятся дубли.

Если вам нужна защита от дублей, обычно делают одно из двух:

* перед `append()` пробегаются по `rules` и сравнивают “сигнатуру” правила (например, `actions + scope + condition`);
* или заводят метку в правиле (если API/схема позволяет) и ищут по ней.

(В текущей версии этого нет.) 

## Безопасность

* **Не коммитьте OAuth-токен** в репозиторий.
* Минимум: храните токен вне репо (например, в локальном файле, переменных окружения или секретах CI).
* Если токен уже попал в git-историю — отзовите его и выпустите новый.

## Ограничения и нюансы

* Условие использует `"$contains": ".<ext>"` по `attach:filename`. Это строковый матч по имени файла, а не анализ содержимого.
* Учитывайте кейсы:

  * `file.EXE` vs `file.exe` (если `contains` чувствителен к регистру, часть файлов может не матчиться).
  * “двойные расширения” типа `invoice.pdf.exe` — текущий матч по `.exe` сработает.
* Правило создаётся как `inbound`. На исходящие (`outbound`) оно не влияет. 

## Troubleshooting

### 1) Ошибка при получении правил (GET не 200)

Скрипт выведет:
`Ошибка при получении правил: <status_code> <body>`

Частые причины:

* Неверный `ORG_ID`
* Неверный/просроченный `TOKEN`
* Недостаточные права токена на endpoint mail routing rules

### 2) Ошибка при сохранении (PUT не 200)

Частые причины:

* API отвергает структуру payload (например, если в текущих правилах есть поля, которые нужно сохранять “как есть”, а API возвращает их в другом виде)
* Превышены лимиты на количество/размер правил (если такие лимиты применяются в вашей конфигурации)
* Конфликт схемы/версии API (реже)

### 3) Правило “не срабатывает”

Проверьте:

* направление `inbound` (это входящие письма) 
* действительно ли имя вложения содержит `.ext` в ожидаемом виде
* порядок правил: раз у правила `terminal: False`, другие правила могут дополнительно менять поведение, если настроены дальше по цепочке
